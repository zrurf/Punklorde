# 重邮智慧体育门户

也是基于标准CAS SSO登录流程，没有中间层，登录流程实现起来基本没啥大问题，只是刷新机制相当脑残，这里记录一下。

## 流程
登录后，业务域同时签发了`access_token`和`refresh_token`，看似是个完善的双token机制，结果`refresh_token`完全无效，代码里完全没有使用`refresh_token`刷新的逻辑，也没有找到刷新的API Endpoint。

**那它的刷新是如何实现的?**

它直接给你原来的登录注销了，连同CAS域签发的`TGT`一起注销，然后重新跑一遍登录逻辑。。。

**最最最离谱的是**，重新登录后，签发的`access_token`和以前的是一模一样的，根据`expires_in`字段推断，只要原来的`access_token`没过期，那么重新即使登录，也是签发以前的`access_token`。

*但凡学过后端也不至于写出这么降智的代码，嫌攻击窗口不够大是吗？*

## 流程优化
我们在刷新时，不要注销CAS域签发的TGT，把它当作`refresh_token`用，在CAS域重新签发`ST`来刷新即可。
