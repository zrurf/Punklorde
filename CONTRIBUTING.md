# 贡献指南 | Contributing to Punklorde

首先，感谢你愿意为 **Punklorde** 贡献代码、文档或建议！我们欢迎任何形式的贡献，但请注意，本项目的性质和许可证决定了我们需要严格遵守一些规则。

在开始贡献之前，请确保你已经阅读并理解了以下文档：
- [README.md](./README.md)：了解项目的基本情况、声明和风险。
- [LICENSE](./LICENSE)：了解代码的使用限制（**非商业学术许可证**）。

## 行为准则 | Code of Conduct

我们致力于维护一个开放且友好的社区。作为贡献者，你需要：
1.  **尊重他人**：使用友好、包容的语言。
2.  **遵守法律**：**严禁**利用本项目进行任何形式的学术欺诈、考试作弊或违反校规校纪的行为。本项目仅用于技术交流与学习。
3.  **责任自负**：如果你提交的代码被用于违规用途，责任由使用者承担，但作为贡献者，你也应意识到代码的潜在影响。

## 如何贡献 | How to Contribute

### 1. 提出问题 | Reporting Issues
- 在提交 Issue 前，请先搜索已有的 Issues，避免重复。
- 使用清晰的标题，并提供详细的环境信息（操作系统、设备型号、Flutter 版本等）。
- 如果是崩溃或 Bug，请附上完整的日志（`logcat` 或控制台输出）。
- **请勿**在 Issue 中询问如何获取构建产物（APK/IPA）或请求预编译的安装包。

### 2. 提交代码 | Pull Requests
我们非常欢迎 PR，尤其是针对现有 TODO 列表或已知问题的修复。

#### 流程
1.  **Fork 本仓库** 到你的账户下。
2.  **创建分支**：
    - `fix/xxx`：修复 Bug。
    - `feat/xxx`：新增功能。
    - `docs/xxx`：修改文档。
    - `refactor/xxx`：重构代码。
3.  **编写代码**：请遵循下方的代码风格指南。
4.  **测试**：确保你的代码在至少一个目标平台（目前主要是 Android）上能够正常运行且不引入新的严重 Bug。
5.  **提交**：请遵循下方的提交消息规范。
6.  **发起 PR**：描述清楚你做了什么、为什么这么做，并关联相关的 Issue（如果有）。

#### 代码风格指南
- **Dart / Flutter**：
    - 严格遵循 `dart format` 格式化。
    - 使用 `flutter analyze` 检查警告，提交前应确保无新增的 `error` 或 `warning`。
    - 命名规范：类名使用 `UpperCamelCase`，变量/方法使用 `lowerCamelCase`。
- **Rust**：
    - 使用 `cargo fmt` 格式化。
    - 遵循 Rust 2024 Edition 的惯例。
- **Go**：
    - 使用 `go fmt` 格式化。
- **注释**：
    - 核心逻辑和对外暴露的 API 必须有清晰的注释。
    - **敏感注释警告**：请勿在注释中留下任何 API Key、密码或特定的校园系统漏洞细节。

#### 提交消息规范
为了保持历史清晰，请使用以下格式：

```
<类型>(<范围>): <简短描述>

<详细描述（可选）>

<脚注（如 Breaking Change）>
```

**类型**：
- `Feat`: 新功能
- `Fix`: Bug 修复
- `Docs`: 文档更新
- `Style`: 代码格式调整（不影响逻辑）
- `Refactor`: 重构
- `Perf`: 性能优化
- `Test`: 测试相关
- `Chore`: 构建过程或辅助工具变动

**示例**：

```
Feat(cqupt): 添加体育打卡的虚拟定位容错机制

修复了当 GPS 信号弱时导致打卡失败的问题，增加了基于基站信息的辅助定位。

BREAKING CHANGE: 需要重新配置百度地图 API Key。
```

### 3. 文档贡献 | Documentation
文档是项目的重要组成部分。如果你发现 `README.md`或 `docs/` 下的内容有误或不清晰，欢迎提交 PR 修改。

## AI辅助编程 | AI-assisted programming
我们不反对AI辅助编程，并且现有的代码中已经存在部分AI编写的模块。但是，在使用、提交AI编写的相关内容时，你需要注意：
1.  请在 PR 中说明。
2.  AI编写的代码要符合项目的结构和命名规范。
3.  AI不得破坏原有的项目结构和命名方式。
4.  AI编写的代码必须完全通过测试才允许提交。

## 适配其他学校 | Adapting to Other Schools
如果你想为你的学校添加适配（课表、签到、校园跑等），请参考 `lib/features/schools/cqupt/` 下的实现。

1.  在 `lib/features/schools/` 下新建你的学校目录。
2.  实现必要的接口（如 `LoginService`, `SportsService` 等）。
3.  在 PR 中说明适配的学校及测试过的功能。

## 关于构建产物 | Regarding Build Artifacts
根据 `README.md` 中的声明：
- 本项目 **不提供** 任何现成的 APK、IPA 或其他构建产物。
- 贡献者 **不应** 在 Issue、PR 或任何渠道中上传或传播构建产物。
- 如果你需要测试，请根据 [构建文档](docs/start/build.md) 自行构建。

## 法律与责任 | Legal & Responsibility
- 你的贡献将被视为同意根据 [LICENSE](./LICENSE) 进行许可。
- 如果贡献内容涉及逆向工程、破解或任何可能违反第三方服务条款的代码，请务必在 PR 中明确说明。
- 项目维护者保留合并或拒绝任何 PR 的权利，特别是那些可能给项目带来法律风险的 PR。

再次感谢你的贡献！但请记得 **保持低调，注意安全**。
